# 全应用弹窗统一迁移设计

## 背景

共享 `MacDialog` 已提供统一的 macOS 窗口控制、标题栏拖拽、收起恢复、最大化、遮罩关闭、Esc 关闭、提交锁定、活动弹窗栈和内容区细滚动条。工作日志以及用户管理、应用管理、数据字典等系统管理页面已经完成迁移，但多数业务应用仍维护页面私有的遮罩、标题栏、关闭按钮、底部取消按钮和滚动样式。

本次将剩余所有模态界面迁移到 `MacDialog`。常驻页面内容、抽屉以外的普通面板、原生提示和星空页面的常驻遮罩面板不属于本次范围。

## 目标

- 让所有应用弹窗统一获得拖拽、收起、最大化、红色关闭、遮罩关闭和 Esc 关闭能力。
- 统一使用共享内容滚动区，删除页面私有的粗滚动条和重复弹窗外壳。
- 移除底部仅用于取消或关闭的按钮，只保留业务主操作。
- 保留所有现有字段、校验、API、提交状态、详情展示和错误处理。
- 对嵌套弹窗、长表单、密码详情和退出确认提供一致且可验证的行为。

## 非目标

- 不修改后端接口、数据库结构或业务规则。
- 不统一页面主体布局、列表、图表或非弹窗表单视觉。
- 不迁移星空页面的常驻 `overlay-panel`。
- 不替换浏览器原生 `alert`、`confirm`，除非当前页面已经使用自定义模态界面。
- 不增加弹窗尺寸记忆、可调整大小或跨会话位置持久化。

## 迁移范围

### 第一批：普通 CRUD

共 5 个弹窗：

- 待办列表：新增/编辑任务。
- 软件仓库：新增/编辑软件、版本。
- 个人账单：新增/编辑账单、年度预算。

### 第二批：详情、安全与首页

共 8 个弹窗：

- 加油统计：新增/编辑加油记录、加油详情。
- 知识库：新增/编辑经验、经验详情与审核操作。
- 密码备忘录：新增/编辑账号、账号详情与密码验证。
- 首页：个人中心、退出登录确认。

### 第三批：复杂长表单

共 5 个弹窗：

- 健康记录：健康指标、医院就诊、报告单。
- 魔兽角色：角色信息、专精键位。

已经迁移的工作日志、用户管理、应用管理和数据字典弹窗不重复改造，只参加最终回归。

## 统一接入方式

每个页面导入并注册 `MacDialog`，使用现有显示状态作为 `v-model`：

- `title` 继续使用当前新增、编辑或详情模式生成。
- `close-disabled` 绑定现有提交、上传或验证中的锁定状态。
- `panel-class` 提供页面级宽度和内容适配，不覆盖共享滚动与窗口行为。
- `cancel` 继续调用页面已有关闭函数，清理附属状态。
- 表单主体放入默认插槽，业务主操作放入 footer 插槽。
- 表单外提交按钮通过稳定的 form `id` 触发原提交函数。

迁移只替换弹窗外壳。字段顺序、`v-model`、条件渲染、原生校验、业务校验、API 参数和成功/失败流程保持不变。

## Footer 规则

- 普通新增/编辑表单只保留保存、更新、确认等主提交按钮。
- 底部“取消”“关闭”“返回”等纯关闭按钮删除。
- 只读详情没有业务操作时不渲染 footer。
- 审核、验证密码、复制键位、确认退出等业务操作继续保留。
- 退出登录确认只保留“退出登录”主操作。
- 随机生成密码、重置、切换模式等会改变内容的功能按钮继续保留在原业务区域。

取消由左上红色按钮、遮罩点击或 Esc 统一处理。移动端隐藏三色控制时，沿用 `MacDialog` 的移动端明确关闭入口，不在业务 footer 重复增加取消按钮。

## 状态与交互

- 提交、上传或验证期间，红色关闭、收起、遮罩、Esc 和主提交按钮按现有状态锁定。
- 绿色最大化在锁定期间仍可使用，不中断请求。
- 收起与恢复保留表单局部状态、未受控输入值、滚动位置和焦点返回目标。
- 嵌套弹窗使用共享活动栈，只有最上层响应 Esc。
- 魔兽角色信息弹窗打开专精键位弹窗时，底层角色表单保持挂载。
- 关闭弹窗时继续执行页面现有清理逻辑，包括下拉状态、密码明文、上传预览、当前详情和临时选择。
- API 失败后继续保留弹窗和用户输入，沿用现有错误提示。

## 样式处理

每个迁移页面删除仅服务旧弹窗的样式：

- `.dialog-mask` 及页面私有遮罩变体。
- 旧 `.dialog`、`.user-dialog`、`.keybinding-dialog` 等外壳尺寸和滚动规则。
- 私有标题栏、标题、副标题和关闭按钮规则中已由 `MacDialog` 提供的部分。
- sticky `.dialog-actions` 和仅为底部取消按钮服务的布局。
- 移动端旧弹窗外壳和遮罩媒体查询。

页面字段网格、详情卡片、密码验证区、上传区、业务分组和内容展示样式继续保留。长表单只允许 `.mac-dialog-body` 滚动，标题栏和 footer 始终固定。

## 分批实施与提交

三批按顺序实施。每批都必须形成独立、可回退的提交，并在进入下一批前完成测试与构建：

1. 普通 CRUD 页建立统一迁移范式。
2. 详情、安全与首页覆盖无 footer、业务操作和确认场景。
3. 复杂长表单覆盖嵌套弹窗、长内容和多业务分组。

若某页面迁移暴露现有业务缺陷，只记录并单独处理，不在弹窗迁移提交中顺带重构无关逻辑。

## 测试与验收

### 结构覆盖

- 新增全应用迁移测试，逐页确认 18 个弹窗使用 `MacDialog`。
- 确认迁移页面不再存在旧 `.dialog-mask` 模板结构。
- 确认业务 footer 不再包含纯“取消 / 关闭”按钮。
- 确认保存、审核、验证、复制和退出登录等主操作仍存在。

### 真实组件行为

每批至少选择一个代表页面做真实挂载测试：

- 新增/编辑打开与详情字段回填。
- footer 外部提交按钮实际触发原提交 API。
- pending 请求期间关闭入口和重复提交被锁定。
- 失败后弹窗、输入和附属状态保留。
- 只读详情无冗余 footer。
- 嵌套弹窗只关闭最上层，底层状态不丢失。

### 自动化与浏览器验证

- 运行全部 Node 单元测试和 Vitest 组件测试。
- 执行 `npm run build`。
- 在桌面、平板和移动端检查弹窗宽度、内容滚动、拖拽、收起、最大化、固定主操作、无横向溢出及控制台错误。
- 回归已经迁移的工作日志与系统管理弹窗，确保共享组件行为未退化。

## 完成标准

- 范围内 18 个弹窗全部使用 `MacDialog`。
- 所有旧业务弹窗遮罩和重复关闭入口已删除。
- 现有业务提交、详情、审核、验证和失败恢复行为保持可用。
- 全量测试和生产构建通过。
- 桌面与移动端真实浏览器验收无阻塞问题。
